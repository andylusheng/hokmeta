# HOKMeta 内容与页面改动约束

Last updated: 2026-07-14

## 目标

这份文档不是讲“怎么写文案”，而是讲：

- 哪些页面允许自动加文字
- 哪些页面只能展示客观数据
- 哪些页面必须人工审核
- 每次开发前先判断“能不能改”，而不是“先改了再解释”

它和 [HOKMETA_TASK_EXECUTION_SOP.md](./HOKMETA_TASK_EXECUTION_SOP.md) 配合使用：

- `SOP` 约束任务流程
- 本文档约束页面内容边界

---

## 一、核心原则

### 1. 页面不是备注区

任何新增文字，都必须先回答：

- 这是给用户看的，还是给站长/开发看的？
- 用户读完之后能多做出什么决策？
- 这段话是否重复了页面上方已经展示的数据？

如果回答不清楚，就不应该加。

### 2. 自动生成的文字必须比手工保守

凡是由代码拼接、规则推断、脚本生成的文字：

- 只能写“客观事实”
- 可以写“状态标签”
- 不允许写“因果判断”

允许：

- `最近 30 天：上升`
- `7 日变化：+0.39 pts`
- `S15：Adjusted Skill Damage`

不允许：

- `通常代表当前版本更适合这个英雄`
- `说明玩家已经学会了正确出装`
- `意味着这个英雄现在更适合上分`

原因很简单：

- 数据不一定支持这个结论
- 用户会把这种话当成官方判断
- 一旦规模化扩散到几千页，可信度会整体下降

### 3. 先定义“页面类型”，再决定“文案权限”

HOKMeta 不是纯博客站，而是：

- 数据库站
- 工具站
- 指南站

这三类页面的文案权限完全不同，不能混用。

---

## 二、页面分级与文案权限

## A 类：数据库 / 数据页

适用页面：

- `/hero/[slug]/`
- `/hero/[slug]/counters/`
- `/items/[id]/`
- `/arcana/`
- `/hero-trends/`
- `/meta-report/`

目标：

- 给用户快速、可信、可操作的答案

允许出现的文字类型：

- 标题
- 一句用途说明
- 数据标签
- 数据来源
- 当前赛季或最近更新时间
- 深链 CTA

限制：

- 不允许自动生成长段“分析”
- 不允许自动生成没有证据的解释
- 不允许用模板话重复数据

例如：

允许：

- `基於官方每日勝率統計。`
- `最近 30 天：上升`
- `S15：無近期官方改動可引用`

禁止：

- `這代表版本節奏更適合這個英雄`
- `玩家已經找到最優出裝`
- `通常說明這個英雄即將成為主流`

### A 类页面的默认规则

#### 1. 首屏说明最多 1 到 2 句

数据库/工具页首屏不是文章正文。

要求：

- 一眼说明这页有什么用
- 不能像博客导语
- 不能写成长解释

#### 2. 数据下方的“趋势/解释”只允许三层

- `数据层`
- `状态层`
- `来源层`

不允许再往下写“推理层”。

标准格式：

- 当前值
- 7 日变化
- 30 日变化
- 状态：上升 / 持平 / 下降
- 版本原因：当前赛季最近官方改动 / 无近期官方改动可引用

#### 3. 右侧解释位优先放来源，不放猜测

像“版本原因”这样的盒子，只能放：

- 当前赛季最近 45 天官方改动
- 无可直接引用改动

不能放：

- 推测性版本结论
- 模板分析

---

## B 类：工具页

适用页面：

- `/tools/`
- `/tools/build-compare/[slug]/`
- `/tools/damage-calculator/[slug]/`
- `/tools/counter-picker/`

目标：

- 让用户完成动作，而不是看长文

允许出现的文字类型：

- 工具用途
- 输入说明
- 输出解释
- 风险提示
- 相关深链

限制：

- 不允许为了 SEO 堆大段说明
- 不允许在工具区块里混进大量教程
- 不允许每个工具页都自动拼出 300 字“分析段”

### B 类页面规则

#### 1. 工具说明优先回答 3 个问题

- 这个工具算什么
- 结果怎么读
- 下一步去哪里看

#### 2. 工具页正文说明不要超过一个短区块

除非是专门的 learn 页面，否则：

- 不写长教程
- 不写策略分析
- 不写内容农场式 SEO 文案

---

## C 类：攻略 / Learn 页

适用页面：

- `/learn/[slug]/`
- 专题 guide
- 长尾 SEO 内容页

目标：

- 承接搜索意图
- 提供完整解释
- 做深度内容和内链枢纽

允许出现的文字类型：

- 结构化正文
- 场景解释
- 版本解读
- 对线 / 团战 / 出装原因
- FAQ

限制：

- 不能写空话
- 不能写 AI 味模板句
- 不能把数据页里那种短句硬拉成长文

### C 类页面规则

#### 1. 攻略页可以解释“为什么”

但前提是：

- 有数据支持
- 有补丁支持
- 有明确对局逻辑支持

#### 2. 攻略页里的判断必须可落地

允许：

- `如果敌方是双前排，这套穿透装更稳`
- `这个英雄在发育路对短手压制更强`

不允许：

- `整体环境更加友好`
- `这个版本节奏更加适配`

除非后面明确解释“为什么”。

---

## 三、文案权限矩阵

| 内容类型 | 数据页 | 工具页 | 攻略页 |
|---|---|---|---|
| 标题/描述 | 允许 | 允许 | 允许 |
| 一句话用途说明 | 允许 | 允许 | 允许 |
| 客观数据摘要 | 允许 | 允许 | 允许 |
| 状态标签（上升/下降） | 允许 | 允许 | 允许 |
| 官方补丁引用 | 允许 | 允许 | 允许 |
| 推测性趋势解释 | 禁止 | 禁止 | 谨慎，需证据 |
| 长段策略分析 | 禁止 | 禁止 | 允许 |
| FAQ | 少量 | 少量 | 允许 |
| 自动生成大段正文 | 禁止 | 禁止 | 原则上禁止，需审核 |

---

## 四、自动文案红线

下面这些句式，默认禁止自动加到页面：

- `通常代表`
- `說明玩家`
- `意味著這個英雄`
- `版本節奏更適合`
- `環境更友好`
- `更適合上分`
- `玩家已經找到最優`
- `現在明顯值得優先投入`

除非：

- 有明确数据规则支撑
- 有人工审核确认

否则一律不写。

---

## 五、趋势类页面的专门约束

适用页面：

- `/hero-trends/`
- `/meta-report/`
- 英雄详情页里的趋势模块
- 趋势类 learn 页面

### 1. 趋势页的说明对象是玩家，不是站长

必须回答：

- 这页能帮玩家做什么决定

不能回答成：

- 我们怎么做数据
- 我们怎么理解样本
- 我们怎么建模

因为这些是内部逻辑，不是玩家决策语言。

### 2. 趋势模块第一版统一极简

英雄详情页趋势模块默认只保留：

- 当前胜率
- 7 日变化
- 30 日变化
- 状态标签：上升 / 持平 / 下降
- 当前赛季最近官方改动 / 无近期官方改动可引用
- guide / counter 深链

不自动生成“趋势判读段落”。

### 3. 趋势类 Learn 页才允许做解释扩展

如果要写：

- 为什么这个英雄最近变强
- 为什么这个英雄最近被低估

这些应该放到：

- `/learn/most-underrated-heroes-this-week/`
- `/learn/highest-win-rate-heroes-this-week/`

而不是塞进每个英雄详情页的趋势盒子里。

## 六、多语言内容约束

### 1. 一个 locale 只要正文没完成，就视为未完成

例如：

- 页面路径是 `zh-TW`
- 但正文还是英文

这种情况不叫“已支持繁中”，只能叫“壳子已切换，正文未完成”。

### 2. 多语言切换器不能先于内容完成度

规则：

- 先完成正文
- 再开放 locale

不能反过来。

### 3. 首页 Hero Splash、趋势说明、工具摘要都算正文

不是只有长文章才算正文。

凡是用户首屏能看到的大段可读文案，都属于正文。

### 4. 多语言完整版本的硬标准

一个 locale 只有同时满足下面条件，才能称为“完整支持”：

- 导航、搜索框、按钮、面包屑、页脚已本地化
- 页面标题、meta title、meta description 已本地化
- 首屏说明、摘要、FAQ、CTA 已本地化
- 列表页和详情页都已本地化
- 动态 slug 页面已生成，并且正文不是英文 fallback
- 同页面语言切换器包含该 locale
- 站内链接保持同 locale 路径
- sitemap / GSC URL 清单只收录已完成页面
- 构建后 `out/{locale}/.../index.html` 存在

只满足其中一部分，只能称为：

- `技术路由已接入`
- `UI 壳已本地化`
- `内容仍为 fallback`

不能称为：

- `ID 已上线`
- `FIL 已上线`
- `小语种已完成`

### 5. Learn 页是最高风险页面

`/learn/[slug]/` 属于长正文页面，不能用 UI 字典和英文 fallback 冒充本地化。

Learn 页开放某个 locale 前，必须检查：

- 文章标题
- 导语
- 正文段落
- 小标题
- FAQ
- 内链锚文本
- 作者/更新时间区域
- 相关 guide / counter 链接

如果任意一项仍是英文正文，该 locale 不允许显示在语言切换器里。

### 6. Fallback 只能用于内部过渡，不能当成上线结果

允许 fallback 的场景：

- 本地开发验证路由
- 明确标注“正文未完成”的内部预览
- 尚未提交 sitemap / GSC 的灰度阶段

不允许 fallback 的场景：

- 对外说某语言已上线
- sitemap 收录
- GSC 提交
- 外链推广
- 首页导航或语言切换器主动引导用户访问

## 七、每次改动前必须先做的页面判断

动任何页面前，先写出：

### 1. 页面类型

- A 类：数据库 / 数据页
- B 类：工具页
- C 类：攻略页

### 2. 这次改动属于哪一类

- 结构改动
- UI 改动
- 数据改动
- 文案改动
- 多语言改动

### 3. 这次是否允许新增说明文字

结论只能是三种：

- `允许，且只能加短说明`
- `允许，但必须人工写`
- `不允许新增解释文案`

如果没先写这三项，不进入编码。

---

## 八、开发时的执行顺序

后续所有 HOKMeta 页面改动，统一按这个顺序：

1. 先判断页面类型
2. 再判断文案权限
3. 再列允许改动的文案范围
4. 再列禁止改动的文案范围
5. 用户确认后再动手

标准模板：

```md
页面类型：
本次改动类型：
允许新增的文字：
禁止新增的文字：
预计修改文件：
验证方式：

如果按这个范围执行，我再开始改。
```

---

## 九、验收清单

涉及文案改动时，提交前必须逐项确认：

- 这段话是给用户看的，不是给开发看的
- 这段话没有重复页面上方已有数据
- 这段话没有无证据因果
- 这段话符合当前页面类型
- 多语言版本正文一致
- `npm run build` 通过
- 若开放新语言，已抽查首页、列表页、详情页、动态 slug 页
- 若是 Learn 页，已确认正文不是英文 fallback
- 若是工具页，已确认工具说明、结果解释、CTA 都是当前 locale

任意一项不满足，都不能算完成。

---

## 十、当前立即执行的默认策略

从现在开始，HOKMeta 默认采用：

### 数据页

- 少写
- 只写用途
- 只写状态
- 不写推测

### 工具页

- 以操作为主
- 文字只服务输入输出理解

### 攻略页

- 深度内容集中放这里
- 解释性文字集中放这里

---

## 十一、这份文档的用途

这份文档从 2026-07-10 起，作为 HOKMeta 页面改动的内容边界约束。

后续如果要改：

- 趋势说明
- 首页摘要
- Hero 页面自动文案
- 工具页说明
- 多语言页面正文

都必须先对照这份文档判断是否允许改。
