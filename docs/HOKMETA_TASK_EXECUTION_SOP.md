# HOKMeta 任务执行 SOP

配套文档：

- 流程约束：本文档
- 页面与文案边界约束：[HOKMETA_CONTENT_AND_PAGE_GUARDRAILS.md](./HOKMETA_CONTENT_AND_PAGE_GUARDRAILS.md)

适用范围：

- HOKMeta 站内功能开发
- SEO 页面改版
- 内容结构调整
- 数据同步后的页面修复
- 外链发布前的站内配套工作

目标：

- 先评估，再执行
- 先收敛范围，再修改文件
- 先定义验证标准，再跑代码
- 避免“先乱改，后解释”
- 避免“只改入口、导航或按钮，就把功能说成完成”

## 零、什么才算完成

HOKMeta 后续所有功能都按“用户完整路径”验收，不按“代码是否写了某个入口”验收。

一个功能不能因为下面这些事情完成就宣称完成：

- 导航栏出现了入口
- 语言切换器出现了按钮
- 路由目录存在
- 静态页面能生成
- 首页能看到某个模块

必须同时满足：

- 用户从首页或相关页面能找到它
- 目标页面能打开
- 页面正文、数据、CTA、内链都与当前功能一致
- 多语言页面的正文语言与 locale 一致
- sitemap / GSC / 内链策略没有被遗漏
- 本地构建和关键页面验证通过

如果只完成了技术入口，汇报时只能说：

- `入口已接入`
- `路由已生成`
- `当前是 fallback 版本`

不能说：

- `已上线`
- `已完成`
- `语言已支持`

除非上面的完整路径验收全部通过。

## 一、固定执行原则

每次新任务开始时，必须先输出一份任务评估卡，再决定是否动手。

禁止直接进入编码的情况：

- 需求边界不清
- 用户明确要求先出方案
- 涉及多个页面或多语言联动
- 涉及 SEO、内链、结构化内容、索引策略
- 涉及数据源、脚本、页面三者联动

可以直接执行的情况：

- 单点小修复，影响范围明确
- 用户明确说“直接做”
- 之前已经确认过同一类模板和步骤

## 二、任务开始前必须输出的评估卡

每次开始前，必须先按下面结构发给用户确认：

### 1. 任务目标

一句话说清楚这次要解决什么问题。

示例：

- 修复 `/meta-report/` 缺少用户可见内链的问题
- 修复 `/zh-TW/learn/...` 正文未本地化的问题

### 2. 问题判断

明确说明当前问题属于哪一类：

- 内容问题
- 结构问题
- SEO 问题
- 多语言问题
- 数据问题
- UI 问题

### 3. 影响范围

必须明确写出：

- 会影响哪些页面
- 会影响哪些语言版本
- 会影响哪些脚本或生成产物
- 是否会影响 sitemap / GSC / public api

影响范围不能只写页面大类，必须列出具体路由族。

示例：

- `/`
- `/hero/[slug]/`
- `/hero/[slug]/counters/`
- `/learn/`
- `/learn/[slug]/`
- `/tools/`
- `/tools/build-compare/[slug]/`
- `/tools/damage-calculator/[slug]/`

如果需求包含“上语言”“上线工具”“优化英雄详情”“攻略中心同步”，必须额外列：

- 导航入口
- 语言切换器
- sitemap
- GSC URL 清单
- 相关内链
- 结构化数据
- 静态导出路径

### 4. 可选方案

至少给 2 个方案：

- 方案 A：最小修复
- 方案 B：结构性修复

每个方案都要写：

- 优点
- 风险
- 工作量

### 5. 推荐方案

明确给出本次推荐做法，并说明原因：

- 为什么现在做这个
- 为什么不做更大的改造
- 为什么这一步最符合当前阶段

### 6. 预计修改文件

只列本次计划会动的文件，不允许模糊描述。

示例：

- `src/views/MetaReportPageView.tsx`
- `src/lib/learn-trend-articles.ts`
- `src/lib/learn.ts`
- `messages/zh-TW.json`

### 7. 验证方式

必须提前写清楚怎么验，不允许“改完再想”。

验证方式至少包括：

- 页面访问验证
- 文案语言验证
- 内链可见性验证
- 构建验证
- 必要时导出产物验证

如果是多语言任务，验证方式必须包含：

- 每个 locale 的首页
- 每个 locale 的目标详情页
- 每个 locale 的列表页
- 语言切换器是否只展示已完成语言
- 正文是否仍有英文 fallback
- `out/{locale}/.../index.html` 是否存在

如果正文还没本地化，不能开放该 locale 的语言入口。

### 8. 等待确认

评估卡最后必须停住，等用户确认后再执行。

标准收尾句：

- `如果按这个方案执行，我再开始改。`

## 三、确认后执行的标准步骤

用户确认后，按下面顺序执行。

### 第 1 步：复查现状

先读代码，再动文件：

- 当前页面组件
- 当前数据来源
- 当前语言映射
- 当前导出路径

目的：

- 防止按记忆改代码
- 防止改错层
- 防止修表面不修根因

### 第 2 步：只改确认范围内文件

执行要求：

- 不额外发挥
- 不顺手改别的页面
- 不把 unrelated data churn 混进来
- 不把“也许以后有用”的改动一起带上

如果执行中发现范围变大，必须暂停并重新发评估卡。

### 第 3 步：边改边对照原目标

每完成一块，都要回看三个问题：

- 这个改动是否直接服务原目标
- 这个改动是否会扩散影响
- 这个改动是否会制造新的验证成本

如果答案不清楚，就不继续扩写。

## 四、执行后的验证标准

改完后，必须按以下顺序验证。

### 1. 页面层验证

检查：

- 页面是否能打开
- 用户能否直接看到关键内容
- 内链是否是“可见且可点击”的
- 繁体页面是否真的显示繁体正文

### 2. 生成层验证

检查：

- `npm run build` 是否通过
- 静态导出页面是否生成
- 对应路由是否在输出里存在

### 3. 内容层验证

检查：

- 标题是否正确
- 描述是否正确
- 正文语言是否一致
- 不允许出现英文壳子包着繁体路径

### 4. 数据层验证

如果涉及 API / JSON：

- 数据文件是否生成
- 字段是否完整
- 是否有明确用途
- 是否和页面或后续工具有连接

## 五、结果汇报模板

执行完成后，必须按下面格式汇报，不允许只说“改好了”。

### 1. 做了什么

只讲用户可感知结果。

示例：

- `/meta-report/` 增加了相关周报入口和英雄操作链接
- `/zh-TW/learn/most-banned-heroes-this-week/` 正文已改为繁体中文

### 2. 怎么验证的

必须写：

- 构建是否通过
- 验证了哪些页面
- 验证了哪些关键文案或链接

### 3. 还剩什么

如果还有没做完的，必须明确列出：

- 未处理项
- 风险项
- 下一步建议

## 六、必须暂停并重新评估的情形

出现以下情况，必须停手，不允许继续往下改：

- 新发现的问题超出原任务范围
- 需要新增 3 个以上文件
- 需要同时改页面、脚本、数据源
- 英文与繁体逻辑不一致
- 发现当前实现路线不是最优
- 验证方式无法证明结果正确
- 发现只改了入口，但正文、内链或相关页面没有覆盖
- 发现 locale 路由生成了，但正文仍是其他语言
- 发现 GSC 已曝光页面没有被纳入本次目标清单

## 七、红线

以下行为禁止再次发生：

- 未评估先动手
- 一边乱改一边解释
- 用“应该没问题”代替验证
- 用户要求先出方案时仍直接编码
- 把无关改动混入当前任务
- 没确认影响范围就改多语言页面
- 只检查首页，不检查相关详情页、列表页和深链页
- 只检查英文，不检查已开放的所有 locale
- 只跑构建，不检查导出的实际 HTML 页面
- 把 fallback 内容包装成“已本地化”
- 把“技术可访问”汇报成“用户完整可用”

## 八、HOKMeta 默认任务模板

后续每次任务开始，默认先发下面这份简版模板：

```md
任务目标：
问题判断：
影响范围：
可选方案：
推荐方案：
预计修改文件：
验证方式：
完成口径：

如果按这个方案执行，我再开始改。
```

`完成口径` 必须明确写：

- 本次算完整上线
- 本次只是技术接入
- 本次只是第一阶段
- 本次不开放给搜索引擎

没有完成口径，不开始执行。

## 九、当前执行约束

针对 HOKMeta 仓库，额外遵守：

- 不使用 `git add .`
- 不把 `hokmeta-public/` 混入主站仓库提交
- 不把数据同步 churn 和页面修复混在一起提交
- 用户未确认前，不推进到 GitHub 提交阶段

## 十、功能完整性审计

每次功能开发完成后，必须做一次功能完整性审计。

### 1. 用户路径审计

回答：

- 用户从哪里进入？
- 用户进入后能完成什么动作？
- 下一步应该点哪里？
- 内链是否能回到英雄页、工具页、攻略页或数据页？

### 2. 页面族审计

回答：

- 是否只改了首页？
- 是否漏了列表页？
- 是否漏了详情页？
- 是否漏了动态 slug 页面？
- 是否漏了 localized mirror？

### 3. 多语言审计

回答：

- locale 路由是否存在？
- 语言切换器是否展示正确？
- 页面标题、说明、正文、FAQ、CTA 是否同语言？
- 是否存在英文 fallback？
- 如果存在 fallback，是否隐藏该 locale 或明确标注未完成？

### 4. SEO 审计

回答：

- sitemap 是否应该包含？
- GSC 提交清单是否应该更新？
- canonical / hreflang 是否合理？
- 页面是否有足够内链，不是孤岛页？

### 5. 汇报审计

最终汇报必须区分：

- `已完成`
- `已接入但未完整本地化`
- `已生成但未开放`
- `还需要人工内容补齐`

禁止用一个“完成了”覆盖所有状态。
