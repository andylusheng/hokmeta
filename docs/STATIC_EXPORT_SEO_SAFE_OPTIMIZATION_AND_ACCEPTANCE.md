# 静态导出体积优化与 SEO 验收文档

更新日期：2026-07-17

## 一、目标

在不降低 HOKMeta 搜索可见性、页面正文质量、结构化数据和内部链接的前提下，继续降低 Next.js 静态导出 `out/` 的体积与 Cloudflare Pages 部署时间。

本轮优化的目标不是“把 HTML 清空”，而是消除重复序列化的运行时数据。

## 二、当前基线

本轮开始前（工具页第一轮优化完成后）的本地完整构建基线：

| 项目 | 当前值 |
|---|---:|
| `out/` 总体积 | 648.6 MB |
| 文件数 | 8,103 |
| 英雄详情四语 | 284.8 MB |
| 装备详情四语 | 130.3 MB |
| 攻略四语 | 78.1 MB |
| API | 7.7 MB |
| Next 静态资源 | 4.6 MB |

已完成的工具页优化证明：

- 改造前 `out/` 约 5.26 GB
- 工具页不再在每个英雄深链重复传递完整英雄、装备数组
- 改造后 `out/` 降至 648.6 MB
- 工具页的独立 title、H1、description、JSON-LD 和内链均保留

本轮执行后的结果：

从最初约 5.26 GB 的错误导出状态到当前 491.6 MB，静态产物已减少约 90.7%。

## 三、SEO 不可触碰项

后续任何体积优化都不得删除或改成仅客户端渲染：

### 1. 页面级 SEO

- `title`
- meta description
- canonical
- hreflang
- robots 规则
- Open Graph 基础信息

### 2. 可抓取正文

- 英雄名称、定位、出装、铭文、技能、克制信息
- 装备名称、属性、被动、适用英雄
- 攻略标题、导语、正文、FAQ
- 数据来源与更新时间

### 3. 结构与发现路径

- H1 与主要 H2
- Breadcrumb
- JSON-LD（Hero / Article / FAQ / WebApplication）
- 英雄页、克制页、工具页、装备页、攻略页之间的可见内链
- 已开放 locale 的对应语言正文与同 locale 内链

### 4. 禁止做法

- 不把英雄详情正文改成浏览器请求后才显示
- 不删除 `index.txt` 或 Next 导出数据文件，除非先证明不会破坏 App Router 导航
- 不为了减小体积删除 FAQ、技能、出装、装备被动等用户答案内容
- 不把 `id` / `fil` 再改回静默英文 fallback

## 四、问题判断

当前体积大，不等于正文一定太长。更可能的原因是：

1. 服务端页面把完整 `Hero` / `GameItem` 对象传给了客户端组件。
2. 客户端组件只使用头像、名称或少量交互字段，但 Next 会序列化整个对象。
3. 同一份数据被 HTML、RSC 导出数据和四个 locale 路由重复保存。

因此优化原则是：

`保留服务端可抓取内容，缩小跨 Server/Client 边界传递的数据。`

## 五、实施阶段

## 阶段 A：装备详情页最小数据化

### 范围

- `/items/[id]/`
- `/zh-TW/items/[id]/`
- `/id/items/[id]/`
- `/fil/items/[id]/`

### 已知风险点

装备详情页会列出“哪些英雄使用此装备”。`HeroAvatar` 是客户端组件；如果直接传入完整英雄对象，会把技能、FAQ、攻略、补丁等不相关字段带入导出数据。

### 改法

- 保持装备正文、被动、使用英雄链接全部服务端输出。
- 给 `HeroAvatar` 只传：`slug`、`name`、`avatar`、`avatarFallback`。
- 不改装备页的 H1、描述、被动、内部链接或 metadata。

### 验收

- 四语装备详情页的 title、H1、装备描述、被动、使用英雄内链都仍存在。
- 代表性装备页 HTML/RSC 体积下降。
- 装备详情四语目录总量与当前 130.3 MB 对比记录。
- `npm run build` 通过。

### 执行结果（2026-07-17）

- 已完成：`HeroAvatar` 仅接收 `slug`、`name`、`avatar`、`avatarFallback`。
- 已验证：四语代表装备页仍包含 H1、meta description、装备正文与英雄内链。
- 目录结果：四语装备详情从 130.3 MB 降至 19.9 MB，减少 110.4 MB（约 84.7%）。

## 阶段 B：英雄详情页 Client 边界审计

### 范围

- `/hero/[slug]/`
- `/zh-TW/hero/[slug]/`
- `/id/hero/[slug]/`
- `/fil/hero/[slug]/`

### 工作方式

先逐个审计英雄详情组件，标记为：

- 服务端纯展示组件：保留完整服务器数据，不做客户端传递。
- 客户端交互组件：定义最小 props，仅传实际使用字段。
- 不需要交互的组件：移除 `use client` 或改为服务端展示。

### 优先审计组件

- 英雄封面和头像
- Build 变体与出装交互组件
- FAQ 折叠组件
- 趋势图组件
- 目录和相关搜索组件

### 验收

- 英雄首屏、出装、铭文、技能、克制、FAQ、趋势、工具深链仍在静态 HTML 中。
- 结构化数据、canonical、hreflang 和时间标签不变。
- 英语和繁中完整正文不减少；ID/FIL 只维持已定义的 locale-ready 内容范围。
- 抽查 Hou Yi、Kaizer、Luban No.7、Garo 四个英雄及四种 locale。
- 英雄详情四语目录总量与当前 284.8 MB 对比记录。

### 执行结果（2026-07-17）

- 已完成：首屏头像只接收显示字段；技能区只接收技能与头像字段；出装交互只接收出装预设、分路和每格出装理由。
- 保留：服务端仍输出英雄首屏、出装、铭文、技能、克制、FAQ、趋势、工具深链与 Schema。
- 已验证：Garo 四语静态页均保留 H1、meta、FAQPage、克制页、伤害计算器与出装比较内链。
- 目录结果：四语英雄详情从 284.8 MB 降至 257.8 MB，减少 27.0 MB。

## 阶段 C：攻略页只做证据型审计

### 范围

- `/learn/[slug]/` 及已开放 locale 镜像。

### 原则

攻略正文本身是 SEO 资产，不能为了体积删除。只检查是否有客户端组件接收整篇文章、完整英雄库或完整装备库；若有，只拆交互 props，不动正文。

### 验收

- 文章 H1、导语、正文、FAQ、作者/更新时间、相关链接仍在 HTML。
- 已开放 locale 不出现英文正文 fallback。
- 页面文本量和主要 H2 没有被削减。

### 执行结果（2026-07-17）

- 已完成审计：`LearnArticleView` 为服务端页面，没有客户端组件接收完整文章、英雄库或装备库。
- 未改正文：攻略四语目录维持 78.1 MB；这是可抓取正文资产，不以体积为理由删除。
- 已验证：英文与繁中 Hou Yi 攻略保留 H1、H2、meta、Article Schema 与英雄/克制内链。

## 补充优化：Counter Picker

这项不属于攻略正文优化，但审计时发现它与旧工具页有相同的问题：四个 locale 均把完整英雄和装备库写入静态产物。

- 改为打开工具后请求 `/api/tools.json`。
- API 只包含 Counter Picker 实际需要的 116 名英雄、109 件装备、头像、胜率字段和 counter/counteredBy 关系。
- 服务端静态页保留 H1、说明、WebApplication Schema、breadcrumb 和 metadata。
- 四语 Counter Picker 从约 19.8 MB 降至 146 KB；`tools.json` 为 371 KB。

## 六、统一验收流程

每个阶段必须独立执行和验收，不能把三个阶段混在一个大改动中。

### 1. 改前记录

- 当前 `out/` 总体积与目标目录体积
- 两个代表页面的 `index.html` / `index.txt` 大小
- 页面关键 SEO 文本清单

### 2. 代码审查

- 列出所有 `use client` 组件
- 列出每一个跨边界 props
- 说明每个 props 保留或缩小的理由

### 3. 构建验证

```bash
npm run build
npm run qa-zh-locale
```

### 4. 静态产物验证

对每个阶段抽查：

- HTML 文件存在
- `title`、H1、meta description 存在
- canonical 与 hreflang 存在
- 主要正文、FAQ、内链存在
- 目标 locale 没有英文 fallback
- 没有完整 `heroes` / `items` 对象被重复序列化到无关客户端组件

### 5. 浏览器验证

在本地或部署预览中检查：

- 首屏正文未丢失
- FAQ 展开、趋势图、出装选择等交互仍能工作
- 工具和相关内链可点击
- 移动端无布局损坏

## 七、完成口径

### 阶段完成

仅当以下全部成立，才能称某一阶段完成：

- SEO 不可触碰项全部保留
- 构建和 locale QA 通过
- 静态页面抽查通过
- 浏览器交互抽查通过
- 体积变化有记录

### 整体完成

当前：`out/` 已从 648.6 MB 降至 491.6 MB。

下一项候选：Build Generator 仍在四语静态页中携带完整英雄资料，约 19.1 MB。它不能直接复用 Counter Picker 的轻量数据，因为它还要保留逐英雄出装顺序与理由；需要先扩充工具 API 的出装字段并补齐等价的理由逻辑，再单独立项实施和验收。

因此本轮停止继续压缩，不为了追求 250-350 MB 而削减服务端正文、语言完整性或关键交互。

## 八、部署前复核

Cloudflare Pages 必须使用：

- Build command：`npm run build`
- Output directory：`out`
- Node.js：22

禁止使用 `npx next build` 作为生产构建命令，因为它会跳过 API 生成与 locale 后处理。
