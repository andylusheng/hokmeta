# HOKMeta 网站整体优化方案与验收总纲

更新日期：2026-07-17  
适用范围：产品、内容、SEO、数据、性能、多语言、推广与发布流程

## 一、站点定位

HOKMeta 的定位不是泛资讯站，也不是只展示英雄资料的静态数据库。

核心定位：

> 面向 Honor of Kings Global 玩家，提供可验证的英雄数据、出装决策、克制关系和实用工具。

站点要同时承担三件事：

1. **搜索答案页**：承接 `Hou Yi build 2026`、`Kaizer best build`、`Garo counter` 等明确搜索意图。
2. **玩家决策工具**：用伤害计算、出装比较、Counter Picker 和每日胜率数据帮助玩家做选择。
3. **主题权威网络**：英雄、装备、攻略、版本、趋势和工具互相连接，形成 HOK Global 的专门站，而不是孤立文章集合。

## 二、总目标与优先级

### 总目标

- 提升 Google 对 HOKMeta 的主题相关性、可信度和可抓取性。
- 让首次搜索进入的玩家能在一页内得到答案，并进入英雄、攻略或工具的下一步路径。
- 将每日同步的官方数据转化为用户可见的趋势功能、内容选题和更新信号。
- 在不牺牲可抓取正文、语言质量和工具体验的前提下，降低静态导出体积和部署时间。

### 优先级顺序

1. **正确性与可抓取性**：数据、页面、语言和部署必须正确。
2. **英雄搜索意图覆盖**：先做有真实搜索需求的英雄与长尾词。
3. **工具回访价值**：工具要能解决实际排位决策，而不是只做展示。
4. **内容深度与内链**：用攻略承担解释，用数据库页承担答案。
5. **外部传播与权威积累**：先做真实社区参与和引用，再谈数量。
6. **商业化**：有稳定搜索与回访后，再评估广告、联盟和合作位。

## 三、页面架构与职责

| 页面族 | 主要任务 | 文案边界 | 必须连接到 |
|---|---|---|---|
| `/` | 说明站点价值，分流到热门英雄、趋势与工具 | 1-2 段清晰介绍，不做长文章 | Heroes、Hero Trends、Tools、Learn |
| `/heroes/`、`/best-heroes/` | 按分路/需求发现英雄 | 以筛选、卡片和短说明为主 | 英雄详情、分路入口 |
| `/hero/[slug]/` | 回答“这个英雄怎么出、怎么打、值不值得练” | 数据页，短而完整，不自动写推测长文 | guide、counter、damage、compare、相关装备 |
| `/hero/[slug]/counters/` | 回答“谁克制谁、选谁应对” | 对局关系和可操作建议 | 英雄详情、Counter Picker、相关攻略 |
| `/items/[id]/`、`/arcana/` | 装备/铭文参考与可用英雄发现 | 客观属性、用途、可见内链 | 使用该装备的英雄和工具 |
| `/tools/*` | 完成一次计算、比较或选人动作 | 仅解释输入、输出和下一步 | 英雄详情、装备、攻略 |
| `/hero-trends/`、`/meta-report/` | 展示可验证的官方每日数据变化 | 当前值、变化、状态、来源；不自动推断因果 | 英雄详情、周报攻略 |
| `/learn/[slug]/` | 承接长尾搜索，解释“为什么” | 人工审核的深度内容 | 英雄页、counter、items、tools |
| `/patches/` | 版本与平衡改动的来源页 | 仅引用可验证的官方改动 | 英雄页、版本攻略 |

## 四、SEO 与内容集群

### 1. 每个重点英雄的固定搜索集群

一个重点英雄至少覆盖以下页面和连接：

```txt
/hero/{hero}/
  -> /learn/{hero}-guide/
  -> /hero/{hero}/counters/
  -> /tools/damage-calculator/{hero}/
  -> /tools/build-compare/{hero}/
  -> 相关装备页
```

英雄详情页负责快速答案；攻略页负责解释出装、连招、分路、常见失误和不同对局选择；工具页负责验证伤害和比较方案。

### 2. 内容生产优先级

先根据 GSC 已曝光词、真实搜索建议和数据热度排期，不按纯强度榜批量生产。

当前优先词形：

- `{hero} build 2026`
- `{hero} best build`
- `{hero} arcana`
- `{hero} counter`
- `how to counter {hero}`
- `{hero} guide`
- `{hero} combo`
- `{role} best heroes`

内容优先级：

1. 已有 GSC 曝光的英雄与词。
2. Top 20 重点英雄。
3. 高出场、高禁用、高胜率变化的英雄。
4. 搜索意图强但资料薄弱的英雄。

### 3. 内容质量规则

- 数据页只写数据、状态、来源和下一步，不自动写“版本更适合”“值得上分”等推测。
- 版本原因只引用近期可验证的官方改动；无可引用改动时明确写“无近期官方改动可引用”。
- 深度解释只放在 Learn 攻略，必须能回到装备、技能、对局或补丁证据。
- 所有攻略必须包含：搜索意图答案、实战场景、内部链接、FAQ、作者/更新时间和数据来源。

## 五、数据产品与工具路线

### 1. 每日数据的用户价值

每日同步字段包括胜率、出场率、禁用率、Tier 与分路。它们不能只留在内部数据库，应转为：

- `最高胜率 Top 10` 与 `最低胜率 Top 10`
- 每条分路胜率前 5
- 单英雄 7 日、30 日胜率/出场率/禁用率趋势
- 每周最高出场、最高禁用、变化最快英雄的攻略选题
- 版本有官方改动时的事实型说明与相关攻略入口

### 2. 工具的产品标准

| 工具 | 用户要完成的动作 | 结果必须包含 |
|---|---|---|
| Damage Calculator | 选择英雄、等级、装备和目标模板 | 实际伤害、公式说明、装备变更结果 |
| Build Compare | 比较两套出装 | 对不同目标模板的结论与英雄深链 |
| Counter Picker | 输入 1-5 个敌方英雄 | 推荐英雄、Ban 优先级、装备应对、团战注意点 |

工具页不能用长篇 SEO 文案填充；深度解释进入对应 Learn 页面。

### 3. 数据与工具验收

- 数据源、同步日期和适用服务器明确可见。
- 客户端工具只加载实际交互需要的数据。
- SEO 标题、H1、说明、Schema 与深链必须留在服务端静态 HTML。
- API 数据字段必须有页面或工具用途，不能只生成无用 JSON。

## 六、性能与部署策略

### 已完成的性能原则

已确认的根因是：完整 `Hero` 或 `GameItem` 跨越 Server/Client 边界，被重复写入多语言静态页面。

已执行：

- 工具深链改为请求轻量 `/api/tools.json`。
- 装备详情中的头像组件只接收显示字段。
- 英雄详情中的头像、技能和出装交互组件只接收最小字段。
- Counter Picker 改为打开后加载轻量工具数据。
- 攻略正文维持服务端输出，不为压缩删除可收录文本。

当前结果：`out/` 已从约 5.26 GB 降至 491.6 MB。

### 后续性能规则

1. 不删除 title、description、canonical、hreflang、JSON-LD、正文、FAQ 或可见内链。
2. 不把英雄/攻略正文改为浏览器请求后才显示。
3. 每次新增 Client Component，都要定义最小 props，不直接传完整 `Hero[]`、`GameItem[]` 或整篇文章。
4. 每次体积优化都要记录改前/改后目录体积，并抽查四语静态 HTML。
5. Cloudflare Pages 固定使用：`npm run build`、输出目录 `out`、Node.js 22；禁止使用 `npx next build`。

### 下一项候选

Build Generator 仍在四语页面中携带完整英雄资料。它需要先设计等价的出装顺序与出装理由轻量数据契约，不能为了减体积丢失功能后直接迁移。

## 七、多语言策略

### 语言优先级

1. 英文：主 SEO 与 Global 玩家入口。
2. 繁体中文：完整可用的第二语言。
3. 印尼语：优先补齐高流量页面与工具。
4. 菲律宾语：采用统一 Taglish 风格，先补齐高流量页面与工具。

### 语言开放规则

一个 locale 只有在以下内容均完成后，才允许出现在面向用户的语言切换器、sitemap、GSC 提交和外链中：

- 导航、按钮、面包屑、页脚。
- Title、description、首屏说明、FAQ、CTA。
- 对应列表页、详情页、动态 slug 页面正文。
- 站内链接保持同 locale。
- 静态导出路径存在。

Learn 长文是最高风险项：标题、导语、正文、小标题、FAQ、内链锚文本、作者/更新时间必须全部本地化。只翻译 UI 壳不算该语言上线。

### 发布顺序

1. 首页、导航、Hero Trends、Tools。
2. Top 20 英雄详情、Counter、对应工具深链。
3. GSC 已曝光英雄与对应 Learn 文章。
4. 版本页与攻略中心。
5. 其余英雄与长尾文章。

## 八、站外推广与权威积累

### 原则

外链不是每天复制同一篇推广文。Medium、DEV.to、GitHub、Reddit 等平台的主要价值是发现、引流、索引线索和可验证的项目存在感，不应被当作“稳定传递高权重”的主策略。

更有效的方向：

- 在 HOK 玩家社区回答具体 build、counter、版本与工具问题。
- 为 Discord 社群、内容创作者、攻略作者提供可引用的公开数据/API。
- 用周报、数据变动、工具方法做可被二次引用的内容。
- 争取游戏 Wiki、攻略站、视频描述、社区资源页、创作者工具列表中的真实引用。

### 发布纪律

- 每篇外链只服务一个具体问题和一个深链，不群发首页。
- 必须包含实测数据、判断方法、局限或玩家问题；禁止“我做了个站，欢迎访问”的空帖。
- 记录发布时间、URL、目标页面、平台反馈、GSC 曝光变化。
- V2EX 未激活前不纳入日常计划。

## 九、运营节奏

### 每日

1. 发布或准备一篇有明确玩家价值的社区内容/外链。
2. 执行 Camp 同步和 D1 快照，检查同步是否成功。
3. 检查 GSC 已曝光词、抓取异常、索引异常与最近更新页面。
4. 从胜率、出场率、禁用率变化中产出 1-3 个候选选题。

### 每周

1. 发布一篇基于数据和实战逻辑的 Meta/英雄攻略。
2. 复盘 Top 20 英雄：内容是否完整、内链是否贯通、FAQ 是否覆盖真实问题。
3. 汇总 GSC：曝光增长词、平均排名 8-30 的词、CTR 偏低页面、已收录但无点击页面。
4. 检查 `out/` 体积、构建时间和 Cloudflare 部署时间。

### 每月

1. 更新核心英雄的 build、counter、patch、author、更新时间。
2. 审核 sitemap、hreflang、robots、canonical 和 GSC 优先 URL。
3. 更新外链资产台账，淘汰低质量重复发布渠道。
4. 评估广告位、联盟合作或数据/API 合作是否已具备流量基础。

## 十、指标体系

| 维度 | 主要指标 | 判断方式 |
|---|---|---|
| 收录 | 已索引 URL、抓取失败、提交后收录时间 | GSC 页面与站点地图 |
| 搜索 | 曝光、点击、CTR、平均排名 | GSC 查询/页面维度 |
| 内容 | Top 20 英雄完整度、guide 覆盖率、内链覆盖率 | 内容台账与静态检查 |
| 工具 | 工具访问、工具到英雄页跳转、重复访问 | Web Analytics 事件 |
| 数据 | 同步成功率、最新快照日期、D1 覆盖英雄数 | 同步脚本与趋势页 |
| 性能 | `out/` 大小、构建时长、部署时长、CWV | 本地构建与 Cloudflare/GSC |
| 多语言 | locale 正文完成率、错误 fallback 数 | locale readiness 清单 |

早期不以点击量单独判断成败。优先看：曝光是否稳定增加、目标长尾是否开始进入前 30、已有曝光词的页面是否持续更新且匹配意图。

## 十一、统一执行与验收流程

任何功能、SEO、内容或多语言任务都必须遵守：

1. 先输出任务评估卡：目标、问题判断、路由族、语言、数据/脚本、方案、文件、验证方式、完成口径。
2. 用户确认后才开始编辑。
3. 只改确认范围；发现范围扩大时暂停重新评估。
4. 构建后必须检查实际导出的 HTML，而不只看代码。
5. 最终汇报必须区分：`已完成`、`技术接入`、`已生成但未开放`、`仍需人工本地化`。

最小验证组合：

```bash
npm run build
npm run qa-zh-locale
```

按任务补充：

- 四语 route 与正文检查。
- sitemap/GSC 清单检查。
- API 契约检查。
- 本地或预览环境交互检查。
- 静态 HTML 中 title、H1、description、Schema、FAQ、内链检查。

## 十二、未来 30 天路线图

### 第 1 周：正确性与抓取基础

- 固定 Cloudflare Pages 构建命令、输出目录和 Node 版本。
- 将本轮性能优化提交并部署，确认线上 `tools.json` 与 Counter Picker 可用。
- 复查 sitemap、GSC 优先 URL 和已曝光英雄页面。
- 补齐 Top 20 英雄的固定页面模板与内部链接。

### 第 2 周：数据驱动内容

- 上线每周高胜率/低胜率/高出场/高禁用的事实型榜单与 Learn 文章。
- 为 GSC 已曝光英雄补齐对应 guide 与 counter 内容。
- 开始 ID/FIL 的 Top 20 英雄与工具页完整正文本地化，不开放未完成 Learn 页面。

### 第 3 周：工具与回访

- 为 Build Generator 设计轻量数据契约，完成独立评估后再实施。
- 追踪工具到英雄页、攻略页的跳转，补齐无下一步路径的工具结果。
- 基于用户常用查询补充伤害计算和 Build Compare 的目标场景。

### 第 4 周：权威与复盘

- 发布一篇有数据来源和方法说明的周度 Meta 报告。
- 向 HOK 社区、创作者和工具目录推广具体数据页面或工具，不重复发首页宣传。
- 用 GSC 对比四周前后：曝光词、前 30 词、CTR 和已收录页面。
- 决定下一个 30 天的英雄内容优先级与语言扩展范围。

## 十三、完成口径

本方案不是“一次性做完”的开发清单，而是 HOKMeta 后续工作的总约束。

每一项只有同时满足以下条件，才可标记完成：

- 用户路径完整。
- 页面正文与数据可信。
- 已开放语言正文真实本地化。
- 站内内链与 sitemap/GSC 没有遗漏。
- 构建、静态产物和必要交互验证通过。
- 指标或验收证据已记录。

任何只完成入口、路由、按钮、技术接入或英文 fallback 的事项，必须明确标为阶段性结果，不能称为完整上线。
